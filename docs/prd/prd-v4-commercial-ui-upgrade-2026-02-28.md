# GZEAMS V4 商业化 UI/UX 进阶升级 PRD

## 1. 文档信息
- 文档版本: `v1.0`
- 日期: `2026-02-28`
- 适用范围: `GZEAMS 前端系统 (Vue3 + Element Plus)`
- 核心目标: 在 V1~V3 的基础上，进一步提升系统的“数据纵深体验”与“微观交互质感”，并确保运行时与设计器的绝对一致性。

---

## 2. 升级背景与总体目标
随着 V2（全局高级视觉）与 V3（Record Profile 档案头与对象标识）的落地，系统已经具备了顶尖商业 SaaS 的初步骨架。V4 阶段的核心目标是解决**“深层数据消费时的疲劳感”**。
具体表现在：长表单滚动困难、数据加载时的顿挫感、复杂表格的操作信噪比过高，以及缺乏动态生命周期追踪。

---

## 3. 核心特性规划

### 3.1 详情页深度解构 (Record Detail Evolution) 🌟 [最高优先级]
**现状**：当前 `BaseDetailPage.vue` 采用单一瀑布流长滚动设计，当对象配置了过多字段或关联子表时，页面高度失控，查阅效率低下。

**解决方案**：引入**双栏标签页 (Tabs) 布局** 或 **锚点电梯导航 (Scrollspy)**。
为了适应最复杂的业务场景，推荐采用 **主内容 Tabs + 右侧边栏** 的布局形态：
- **左侧主区域 (Main Content)**：使用大 Tabs 切换不同的信息维度，例如 `[详细信息] [相关设备] [生命周期]`。这彻底消灭了无止境的纵向滚动。
- **右侧边栏 (Sidebar)**：固定展示核心高频字段（如：状态标识、责任人、价格摘要），这些信息在切换 Tabs 时始终可见。

⚠️ **关键约束：与“布局设计器 (Layout Designer)”渲染效果的一致性**
- **设计器所见即所得**：若详情页转变为 Tabs 布局，现有的 `WysiwygLayoutDesigner` 必须同步升级。设计器中的画布区必须能真实渲染出 Tabs 结构，允许管理员在不同的 Tab 之间拖拽字段板块。
- **元数据同构**：运行时 (`DynamicDetailPage`) 与设计时 (`LayoutDesigner`) 必须消费同一套被扩展过的 `LayoutConfig` JSON 结构（即 Schema 需要从原来简单的 `sections: []` 扩展支持 `tabs: [{ name: '', sections: [] }]` 层级结构）。
- **组件复用**：设计器中渲染区块的底层组件，必须与 `BaseDetailPage.vue` 中使用的区块组件为同一套纯渲染函数/组件，彻底杜绝“配出来是一个样，跑起来是另一个样”的情况。

### 3.2 骨架屏加载 (Skeleton Loading)
**业务价值**：消除硬切页面或全屏遮罩 Loading 带来的焦躁感。
**具体实现**：
- **列表页**：在拉取当前页数据时，不阻断 UI，而是渲染 10 行等高的灰色斑马纹骨架条。
- **详情页**：在拉取具体 Record 数据时，渲染贴合表单 Section 结构的骨架块（如 2列/3列表单的占位符）。

### 3.3 数据表格的极致质感 (Advanced Data Grid)
**业务价值**：提升高密度数据区的阅读信噪比。
**具体实现**：
- **等宽数字对齐 (Tabular Nums)**：针对所有 `money`, `number`, `date`, `datetime` 类型的列，默认强制**右对齐**，并全局注入 `font-variant-numeric: tabular-nums;` CSS 属性，确保上下行的数字小数点完美垂直对齐。
- **行级悬浮操作 (Hover Actions)**：将目前固定占据表格右侧最宽列的“操作栏”隐藏。只有当用户的鼠标 `hover` 到表格特定行时，该行的最右侧才浮现出快捷操作图标（如编辑、删除、分配），使表格整体视觉极其清爽。

### 3.4 业务时间轴 (Activity Timeline)
**业务价值**：打破静态数据展示，提供动态溯源能力。
**具体实现**：
- 在详情页（可作为独立 Tab 或右侧边栏的一部分）增加 Timeline 组件。
- 展示对象的核心生命周期节点（例如：`2026-02-28 10:00 创建资产 -> 14:00 状态变更为待领用 -> 16:00 分配给 张三`）。

### 3.5 情境化空状态 (Illustrated Empty States)
**业务价值**：化解“无数据”时的尴尬与茫然，提供下一步行动指引。
**具体实现**：
- 替换现有的“暂无数据”文案，引入 3-4 张精细的 SVG 商业插画。
- 区分场景反馈：
  1. **初始化空状态**：列表真实为空。插画提示“开始构建您的数据”，附带主要操作按钮。
  2. **搜索无果状态**：用户使用了过滤器。插画提示“未找到匹配项”，附带“清空筛选条件”的文字链。

## 5. 详细需求指引

### 5.1 详情页双栏布局与定位属性 (Page Layout & Section Position)
- **核心逻辑**：布局系统 Schema (`LayoutConfig`) 扩展支持 `position` 属性。允许渲染引擎（即 `BaseDetailPage.vue`）根据 `position: 'main' | 'sidebar'` 将 Section 动态分发至左侧宽幅主区或右侧定宽边栏。
- **向后兼容性**：若旧版 Schema 未包含 `position` 字段，默认全部退化为 `main`（主内容区瀑布流），确保存量界面的平滑过渡。
- **设计器交互**：在 `LayoutDesigner` 中需明确提供针对板块“位置(Position)”的设置面板或拖拽热区，允许以 WYSIWYG 方式编排主次视觉。

### 5.2 极致质感的微交互 (Refined Micro-Interactions)
- **悬浮操作列 (Hover Actions)**：列表组件需对 Table 进行样式覆盖，在行级别的 `<tr>` 的 `hover` 伪类触发时，才渐显出原先一直占据空间的“编辑/删除”按钮组。要求过渡平滑，并确保点击热区不受影响。
- **数字等宽排版 (Tabular Nums)**：针对所有包含金额 (`money`)、数量 (`number`) 的数据列，统一挂载全局 CSS 类或自定义样式，强制应用 `font-variant-numeric: tabular-nums;`，避免不同位数的数字因字体度量不对齐产生的锯齿感。

### 5.3 骨架屏与空状态 (Skeleton & Empty States)
- **全局骨架屏策略**：禁止一切影响用户注意力的硬性白屏和模态遮罩。加载对象详情时，挂载预设的表单/列表骨架 `ElSkeleton`，直至实际数据被安全渲染完毕 (Suspense/onMounted)。
- **插画空状态**：使用 3~5 款统一风格的商务风插画（Svg）。封装为通用的 `<BaseEmptyState />` 组件，接收 `type` 属性即可自动适配不同的空白场景，并支持通过 slots 插入具有号召性的“Create New”按钮。

## 6. 验收标准 (Acceptance Criteria)

1. **设计器/运行时 100% 同构映射**：在 `WysiwygLayoutDesigner` 配置完毕后提交的布局配置，到 `BaseDetailPage` 中的实际渲染效果，在 DOM 排序与视觉骨架上必须完全等价且不引发控制台警告。
2. **侧边栏吸顶与双区滚动互不干扰**：当 `main` 区域内容极长时往下滚动浏览，`sidebar` 区域若较短，应当启用 `position: sticky` 吸附于视口顶部，不会被主屏带着滚出视野外。
3. **历史数据 0 报错 (Zero-Error for Legacies)**：更新发布后，过去未被重新生成或维护的 LayoutConfig 仍能通过优雅降级正常渲染，无任何崩溃现象。

---

## 7. 开放性问题与隐患 (Open Questions & Risks)

- **Timeline 接口负荷预估**：Phase 4 中提到的“Timeline 日志”，若是高频流转业务对象操作日志，列表可能会极大。需与后端约定好支持分页（Cursor-based Pagination）或采用按需懒加载的模式提取活动数据。
- **组件嵌套地狱**：随着布局层级变深（如未来的 `Layout -> Main -> Tabs -> TabPane -> Section -> Field`），响应式深度监听器（watch/computed）与组件参数透传 (Prop drilling) 的开销可能急剧上升，必要时需考虑 `provide/inject`，或利用 Pinia 进行配置状态的扁平化托管。
