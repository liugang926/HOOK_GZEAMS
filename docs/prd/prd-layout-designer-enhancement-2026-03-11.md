# 布局设计器增强 PRD（修订版）

## 1. 文档信息

- 版本: 1.1
- 日期: 2026-03-11
- 适用范围: `frontend/src/components/designer`、`frontend/src/components/common`、`frontend/src/components/engine`、`backend/apps/system`
- 参考产品: FormCreate Designer
- 修订目的: 基于 NEWSEAMS 当前代码现状，重新界定已具备能力、真实缺口、优先级和交付顺序，避免重复立项

---

## 2. 背景与结论

NEWSEAMS 当前的布局设计器已经不是“从 0 到 1”的状态，而是一个已有较完整骨架的 WYSIWYG 设计器。当前问题不在于缺少设计器框架，而在于以下三类 gap:

1. 设计器侧已经支持的能力，没有在运行时形成完整闭环。
2. 文档中的部分“待开发功能”实际上已经部分落地，但未接线、未统一、或未达到可交付标准。
3. 数据模型在 `designerTypes`、`metadata`、运行时字段模型之间存在命名和语义分裂，后续继续叠加功能会放大维护成本。

本次增强的核心目标不是“功能数量最大化”，而是优先完成“模型统一 + 运行时生效 + 关键体验补齐”。

---

## 3. 当前基线

### 3.1 已具备能力

- WYSIWYG 设计器主框架已完成: Toolbar、FieldPanel、Canvas、PropertyPanel、HistoryPanel 已具备。
- 字段画布已支持真实控件预览基础能力: `DesignerFieldCard` 已复用运行时 `FieldRenderer`。
- 属性面板已支持折叠分组基础结构。
- 字段属性已支持 `required`、`readonly`、`min_length`、`max_length`、`regex_pattern`、`validation_message` 等配置入口。
- 画布已支持撤销/重做、Section 拖拽容器、画布缩放/视口切换、历史记录、草稿与持久化。
- 画布底部统计栏 UI 已存在，但上层未完整接线。
- 跨 Section 拖拽高亮样式和事件已存在。
- 单布局模型已存在，`readonly/detail` 与共享 edit layout 的保存逻辑已存在。

### 3.2 未完成闭环

- 运行时详情页尚未根据布局字段配置自动生成 `formRules`。
- `visibilityRule` 已出现在设计器类型中，但未形成设计器编辑器、运行时求值、发布回读的完整链路。
- 设计器属性模型和运行时字段模型存在 camelCase / snake_case 混用。
- 画布统计栏、未分配字段进度、拖拽插入指示等体验项存在样式或局部实现，但尚未形成完整可验收功能。

### 3.3 当前不建议重复立项的内容

- “字段真实预览”不能再按全新功能立项，应按“预览补差”执行。
- “属性面板折叠分组”不能再按全新功能立项，应按“分组内容优化与统一”执行。
- “跨 Section 高亮”不能再按从零开发估算，应按“补齐验收与回归测试”执行。
- “画布统计栏”不是新设计，而是接线型工作。

---

## 4. 目标与非目标

### 4.1 本次目标

- 让设计器中可编辑的核心字段属性在运行时真正生效。
- 统一布局字段配置模型，减少后续继续叠加功能时的兼容成本。
- 提升设计器关键交互反馈，使其达到“可稳定使用”的商业级内部工具标准。
- 在不破坏单布局模型的前提下，明确 `edit / readonly / detail` 共享与覆盖语义。

### 4.2 本次非目标

- 不追求一次性对标 FormCreate 的全部高级能力。
- 不在本期引入新的独立规则体系，避免与现有 `Business Rule / Rule Engine` 重复建设。
- 不在本期扩展 AI 助理、样式可视化编辑器、移动端专用渲染器。
- 不重写现有设计器主框架。

---

## 5. 关键产品约束

以下语义必须在开发前锁定，否则会导致实现反复:

1. 隐藏字段是否保留已有值。
2. 隐藏字段是否参与校验。
3. 只读字段是否仍允许默认值注入。
4. `readonly/detail/search` 是否共享同一字段规则，还是仅共享布局结构。
5. 设计器内“预览态”和运行时“只读态”是否允许表现不同。

建议默认策略:

- 隐藏字段保留值，但不显示。
- 隐藏字段不参与前端必填校验。
- 只读字段允许展示默认值，但不允许用户编辑。
- 布局结构共享，行为规则允许按 `view_mode` 覆盖。

---

## 6. 方案原则

### 6.1 模型先于交互

先统一字段配置模型，再补交互；否则每新增一个开关都需要同时兼容多种命名与落盘格式。

### 6.2 运行时闭环优先

优先做“保存后在详情页能生效”的能力，再做右键菜单、双击快编等提效交互。

### 6.3 复用已有规则能力

条件显隐优先复用现有 `ConditionBuilder` 和后端 `Rule Engine` 思路，不新增完全独立的第三套规则描述。

### 6.4 单布局模型不被破坏

任何新增能力必须兼容当前 shared edit layout / readonly override 机制。

---

## 7. 修订后的功能规划

## Track A - 配置模型统一与运行时闭环（P0）

目标: 把设计器里已经可编辑的字段行为真正打通到运行时。

### A1. 统一字段配置命名与落盘格式

- 涉及文件:
  - `designerTypes.ts`
  - `types/metadata.ts`
  - `designerLayoutAdapters.ts`
  - `useDesignerPersistenceActions.ts`
- 改动:
  - 明确布局持久化主格式。
  - 对 `min_length/max_length/regex_pattern` 与 `minLength/maxLength/regexPattern` 建立统一 adapter。
  - 避免新增 `LayoutField.validation` 这种新的第三种结构。
- 验收:
  - 保存、发布、回读后字段配置不丢失。
  - camelCase / snake_case 可兼容，但内部处理路径统一。

### A2. 运行时表单规则生成

- 涉及文件:
  - `useBaseDetailPageFields.ts`
  - `BaseDetailPage.vue`
  - 相关详情页接入点
- 改动:
  - 根据布局字段的 `required`、长度限制、正则、数值范围生成 `formRules`。
  - 隐藏字段自动跳过校验。
  - 只读字段默认不触发用户输入校验。
- 验收:
  - 设计器配置的 required/min/max/pattern 在 edit runtime 可实际校验。
  - 错误提示可使用自定义 `validation_message`。

### A3. 条件显隐闭环

- 涉及文件:
  - `FieldPropertyEditor.vue`
  - `useBaseDetailPageFields.ts`
  - `BaseDetailSectionCard.vue`
  - 可选复用: `ConditionBuilder.vue`
- 改动:
  - 先支持单条件显隐闭环，字段 A 变化时字段 B 实时显隐。
  - 第二阶段再视情况升级为复合条件或 JSON Logic。
- 产品建议:
  - 本期不另起新规则中心，优先做布局级轻量规则。
  - 但数据结构应与现有规则体系可映射。
- 验收:
  - 可配置显示条件。
  - 运行时能实时显隐。
  - 隐藏字段不参与前端校验。

建议工期: 6-8 人天

---

## Track B - 设计器已有能力补差（P1）

目标: 将“已部分存在”的能力补齐到可交付状态。

### B1. 字段真实预览补差

- 不再作为从零功能开发。
- 重点处理:
  - disabled 态样式一致性
  - 不可预览字段的优雅降级
  - reference / related_object / sub_table 的设计态显示标准
- 产品修订:
  - reference 字段验收标准调整为“可识别的只读预览”，不强制要求 disabled picker 外观。

### B2. 属性面板分组优化

- 保留当前 `basic/display/validation/advanced` 四段式结构。
- 不建议退化成 PRD 初版中的三组结构。
- 本期重点:
  - 分组顺序优化
  - 文案统一
  - 仅显示当前字段类型相关配置
  - 系统字段 readonly 锁定能力

### B3. 统计栏与未分配字段提示

- 涉及文件:
  - `DesignerCanvas.vue`
  - `WysiwygLayoutDesigner.vue`
  - `DesignerFieldPanel.vue`
- 改动:
  - 接通统计栏字段数/必填数/Section 数。
  - 字段面板展示“已分配 X/Y”进度。
  - 未分配字段在面板中高亮。
- 验收:
  - 数据与画布内容一致。
  - 新增/删除/撤销后统计实时更新。

建议工期: 3-4 人天

---

## Track C - 拖拽反馈增强（P2）

目标: 提升拖拽可感知性，但仅做高价值反馈项。

### C1. 插入位置指示

- 当前已有 `Sortable` 动画和部分样式基础。
- 本期聚焦:
  - 明确插入线或占位样式
  - 在跨容器拖拽时位置反馈稳定

### C2. 拖拽源预览

- 自定义 ghost 仅在桌面端启用。
- 移动端保留默认拖拽反馈，避免兼容性风险。

### C3. 放置动画接线

- 样式类已具备时，优先补逻辑触发而非重写样式。

### C4. 跨 Section 高亮验收

- 现有逻辑保留，补边界清理和回归测试。

建议工期: 2.5-4 人天

---

## Track D - 高级交互增强（P3）

目标: 在核心闭环完成后，再做提效交互。

### D1. 双击快编 Label

- 成本低、收益高，可优先于右键菜单落地。

### D2. 右键菜单

- 仅保留高频操作:
  - 删除
  - 设为必填
  - 设为只读
  - 移动到其他 Section

### D3. Section 排序

- 与字段拖拽不是同一复杂度。
- 涉及容器层级、历史记录、回滚、空容器位置问题。
- 应独立估时，不建议和字段 DnD 优化打包。

建议工期: 3-5 人天

---

## 8. 优先级与实施顺序

建议按以下顺序执行:

1. Track A
2. Track B
3. Track C
4. Track D

原因:

- Track A 解决“能不能真正用”的问题。
- Track B 解决“已有能力看起来和行为是否一致”的问题。
- Track C 解决“拖起来是否顺手”的问题。
- Track D 解决“高手用起来是否更快”的问题。

## 11. 技术风险

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| 字段配置模型多套命名并存 | 保存回读不一致，运行时不生效 | 先做统一 adapter，再继续叠加功能 |
| 单布局模型下 readonly/edit 语义冲突 | 发布后表现异常 | 明确 `view_mode` 覆盖策略，补回归测试 |
| 条件显隐与现有 Business Rule 重复 | 后续维护成本高 | 设计成可映射结构，避免双真相源 |
| 自定义 ghost/插入线影响拖拽稳定性 | 拖拽体验退化 | 桌面端增强，移动端降级 |
| 运行时校验只做前端未做后端同步 | 前后端行为不一致 | 后端校验策略单独立项或至少做契约说明 |

## 12. 验收标准

### 核心验收

- 设计器中可配置的 required/readonly/长度/正则规则，在运行时可真实生效。
- 隐藏规则在运行时可根据表单值实时切换。
- 隐藏字段不参与前端必填校验。
- 保存、发布、刷新后配置不丢失、不串模式。

### 体验验收

- 画布统计栏数字正确且实时刷新。
- 字段面板能识别未分配字段。
- 拖拽时有明确的位置反馈和目标反馈。
- 常见字段在设计器中表现接近运行时。

### 回归验收

- 撤销/重做不破坏新增字段行为配置。
- 发布、回滚、复制布局后新增配置保持一致。
- `edit / readonly / detail` 共享布局路径不被破坏。

---

## 13. 本次修订结论

本 PRD 修订后的核心判断如下:

- 当前项目已经具备成熟设计器骨架，应停止把已有能力按全新功能重复立项。
- 本阶段真正的主战场是运行时闭环与模型统一，而不是继续横向堆 UI 功能点。
- 建议先完成 `P0 + P1`，再评估是否继续投入 `P2 + P3`。

这将比原始版本更贴合 NEWSEAMS 当前代码状态，也更适合作为真实排期和开发拆分依据。
