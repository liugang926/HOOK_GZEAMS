# PRD：详情页布局引擎与设计器排版优化

## 1. 背景与目标
**背景**：
当前系统中，对象详情页的字段排版存在杂乱、不对齐、高度错位等体验问题。经前端代码排查，根本原因在于**布局设计器**依赖 Element Plus 的 24 栅格系统（Flexbox `<el-row>`/`<el-col>`），而**运行时详情页**使用的是纯 CSS Grid（`display: grid`）。这种布局引擎的割裂导致了典型的“所见非所得”问题；同时，全局硬编码的 120px 标签宽度以及 Grid 的 `row dense` 特性加剧了长文本、不同高度字段混合排列时的错乱感。

**目标**：
1. **统一布局引擎**：彻底消除设计态与运行态的 DOM/CSS 差异，实现100%的“所见即所得”。
2. **提升排版灵活性**：引入更加现代化的标签对齐方式与动态宽度支持，适应复杂业务表单。
3. **增强排版控制力**：允许用户显式控制字段换行、独占行及空位占位，杜绝非预期的“见缝插针”乱序排版。
4. **提升设计器体验**：升级设计器操作逻辑，并增加多端（移动端）视图预览。

---

## 2. 功能需求说明

### 2.1 统一视图渲染层 (Unified Layout Engine)
- **需求描述**：将布局设计器画布 (`DesignerCanvasFieldRenderer`) 中的 `<el-row>` / `<el-col>` 废弃，重构为与运行时详情页 (`DetailSectionRenderer`) 完全一致的纯 CSS Grid 结构。
- **关联组件**：`BaseDetailPage.scss`, `DetailSectionRenderer.vue`, `DesignerCanvasSectionRenderer.vue`。
- **业务价值**：消除设计器内部和运行时的像素级偏差，用户在设计器看到的高度、宽度与对齐方式，即是最终用户看到的真实效果。

### 2.2 标签对齐与宽度优化 (Flexible Label Placement)
- **需求描述**：
  1. **支持上下对齐 (Top Label)**：在设计器的 Section（区块）级属性和全局级属性中，新增“标签排版 (Label Position)” 选项，可选值为 `[左右对齐 (Left), 上下对齐 (Top)]`。
  2. **解除硬编码限制**：废除 CSS 中固定的 `var(--gzeams-detail-label-width, 120px)`，允许在设计器中对 Label 宽度进行调整（例如提供 100px, 120px, 150px, Auto 等选项）。
- **业务价值**：上下对齐模式对长字段名和多语言容错率极高，能瞬间大幅提升拥挤表单的整洁度。

### 2.3 占位符与独占行列规则 (Empty Space & Full-width)
- **需求描述**：
  1. **强制独占行**：为富文本、大文本、子表单、图片上传等天然需要宽度的组件，提供属性配置“独占一行 (Full Width)”，在底层 CSS 中渲染为 `grid-column: 1 / -1`。
  2. **空白占位符组件 (Empty Placeholder)**：在设计器左侧的组件物料区，新增一个纯用于排版的“空白占位”组件。允许用户将其拖入网格中占据一个坑位，阻止后续字段因 `grid-auto-flow: row dense` 向上乱窜。
- **业务价值**：赋予实施人员显式控制页面流的手段，彻底解决排版串行问题。

### 2.4 设计器移动端预览 (Mobile Preview)
- **需求描述**：
  1. 在 `WysiwygLayoutDesigner` 的 Toolbar 中新增视口切换按钮（PC/Mobile）。
  2. 切换至移动端时，画布容器固定为常见的手机宽度（如 375px 或 390px），并应用移动端媒体查询样式。
- **业务价值**：使管理员能在保存发布前，确认移动端单列排版下长文本字段是否会溢出。

## 3. 设计器关联调整机制

由于底层布局机制从基于 Flexbox 24栅格 的方案向 CSS Grid（按语义化列数 `columns`）迁移，必须处理好历史数据的兼容与拖拽逻辑的转换：

1. **拖拽缩放 (Resize) 交互重构**：
   原有缩放手柄是计算像素偏移对应多少个 `span24`。现需重构为：计算鼠标拖过多少个 CSS Grid 的网格轨道 (Tracks)，更新配置数据中的 `colSpan` 属性（如 1列, 2列, 3列），而不是 12/24、8/24。
2. **数据模型向下兼容 (Legacy Span Fallback)**：
   在 `semanticGrid.ts` 中继续保留并强化向后兼容逻辑。当读取旧版 Layout JSON 时，若发现基于 24等分 的 `span`（例如 `span: 12`），按照所在 Section 的 columns 配置（例如 2 列），自动换算为语义化 `colSpan: 1` 并在展示和运行时适配。
3. **“网格概念”心智切换**：
   在右侧属性面板中，将字段的宽度选项从“占N个格(总共24)”改为清晰的“占 N 列（当前区块共 M 列）”。

---

## 4. 非功能性需求 (NFR)
1. **性能**：大量字段重构为原生 Grid 布局后，取消了冗余的 DOM 嵌套，首屏渲染性能不可低于现有版本。
2. **兼容性**：支持最新版 Chrome, Edge，响应式布局必须兼容现有 iOS 和 Android 移动端框架（Vant等如果关联使用）。
3. **平滑过渡**：现有数据库中已保存的数十个页面的 `layoutConfig` JSON 数据必须能够无缝渲染，不能出现因升级此特性导致的白屏或崩溃。

## 5. 验收标准 (Acceptance Criteria)

### AC 1: 渲染一致性 (100% WYSIWYG)
- **前置条件**：打开任意包含文本、时间、富文本和图片的混合表单设计器。
- **操作**：在画布中查看字段排列；点击保存并在详情页（运行时）查看实际呈现。
- **预期结果**：运行时的字段宽高、标签与内容框的对齐基线、换行位置与设计器中的预览表现 **100% 一致**。设计器 DOM 检查应确认已使用 `.detail-canvas-grid` 和 `.field-item` 的 Grid 结构。

### AC 2: 标签排版调整
- **前置条件**：进入设计器，选中某个 Section 区块。
- **操作**：将该区块的“标签位置”设置为“上下结构 (Top)”。
- **预期结果**：该区块内所有字段的 Label 移动至内容框上方；长文本 Label 不折行；运行态同步生效。

### AC 3: 独占行与防乱序测试
- **前置条件**：在一个 2 列布局的区块中，第1个字段高度为 40px，第2个字段（多行文本）高度为 100px。向第3个位置拖入一个普通文本。
- **操作 1**：为第2个字段勾选“独占一行”。
- **预期结果 1**：第2个字段横跨两列，第3个字段被正常挤到下一行。
- **操作 2**：取消独占一行，在第1个字段下方（即第3个坑位，由于第2个字段高，此处会留白）拖入“空白占位符”。
- **预期结果 2**：空白占位符填满该坑位，下方的字段不会因 `row dense` 特性错误地浮上来。

### AC 4: 拖拽 Resize 功能验证
- **前置条件**：在一个 3 列布局的区块中拖入一个字段。
- **操作**：拖动字段右侧的 Resize Handle 调整宽度。
- **预期结果**：宽度只能在“占1列”、“占2列”、“占3列”三个状态间吸附跳变，属性面板中的数据同步更新为正确的 `colSpan`。

### AC 5: 历史数据兼容性 (Regression Test)
- **前置条件**：准备一份旧版导出（基于24栅格体系 `span: 12`, `span: 8`）的 `layoutConfig.json` 样例数据。
- **操作**：直接使用系统加载该数据并进入运行时详情页。
- **预期结果**：页面无报错加载，旧的 `span: 12` 在双列布局下被正确解析为各占一列，排版无崩溃。

### AC 6: 移动端视图预览
- **前置条件**：点击设计器顶部 Toolbar 的“移动端预览”按钮。
- **操作**：观察画布变化。
- **预期结果**：画布宽度收缩至移动端尺寸，不论设置为几列，所有字段自动响应折叠为单列 (1 column) 排布，且无横向滚动条溢出。
