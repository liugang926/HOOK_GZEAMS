## 2. 前后端完整度现状对比
基于后端已有的 20 个 App (业务逻辑与模型、测试用例、API) 及数据库表结构，与前端现有视图 (`frontend/src/views`) 及前端路由 (`router/index.ts`) 的深入对比，整体存在明显的**前端功能漏斗**。

后端功能覆盖面：★★★★★ (核心流程资产、耗材、盘点、保险、生命周期、流程引擎、消息等底层模型与接口基本齐备)
前端视图覆盖面：★★★☆☆ (依赖统一的 `objects/:code` 支撑了大量常规表单，但缺乏定制化场景与复杂关联场景的组件)

由于前端页面的滞后，系统暂无法发挥其真正的生命周期闭环和交互能力，亟需重点突击前端缺失页面。

---

## 3. 前端缺失页面与功能梳理 (Gap Analysis)

通过对比，梳理出前端当前缺失的核心业务页面、组件及交互形态：

### 3.1 资产生命周期 (Lifecycle) —— 缺口极大
**后端模块:** `lifecycle` App
**现状分析:** 虽然底层有支持，但前端路由中心没有任何关于 `PurchaseRequest`, `AssetReceipt`, `Maintenance`, `DisposalRequest` 的专属暴露或特殊交互。
**必须补齐功能:**
1. **采购申请与验收入口**: 需建立支持添加明细项的“单据类”表单引擎 (主表+子行的模式)，完成采购动作到入库动作的状态流转闭环。
2. **报废与维护工单面板**: 提供带审批流审批时间轴的维修工单页与报废处置页。

### 3.2 消息与通知中心 (Notifications) —— 完全缺失
**后端模块:** `notifications` App
**现状分析:** `router` 没有任何通知面板的引入，在流程流转、盘点任务指派、租约到期等状态下，用户无法感知到系统的即时触达。
**必须补齐功能:**
1. **全局通知铃铛组件 (Header Bell)**: 在 `MainLayout.vue` 置入，轮询或 WebSocket 接收未读红点。
2. **通知中心面板 (Notification Center)**: 站内信的独立阅读页面，包含各类业务信息的超链接以直接跳转源对象详情页。
3. **偏好与模板配置页**: 管理员与用户的通知触发机制接线板界面。

### 3.3 保险与租赁 (Insurance & Leasing) —— 仅停留于基础路由
**后端模块:** `insurance`, `leasing` App
**现状分析:** `router` 中仅配置了 legacy alias 到统一动态路由，这仅保障了能打开列表与简单平铺式的详情页，无法满足保险、合同等重数据厚交互场景需求。
**必须补齐功能:**
1. **嵌套层级展示能力**: 租赁合同详情必须通过多 Tab，切分呈现“租约基础信息”、“租金打款计划”、“绑定的资产清单”。
2. **特定视图日历聚合**: 保费付款与续保、租金节点需一个专属的图表/工作流日历视图。

### 3.4 登录集成与组织架构 (SSO / Organizations) —— 深度不够
**后端模块:** `sso`, `integration`, `organizations` App
**现状分析:** SSO 无页面化配置手段。组织树呈平铺级展现。
**必须补齐功能:**
1. **SSO 参数与日志面板**: 企微、飞书接入及用户导入记录界面。
2. **拖放式(Drag & Drop)组织树**: 将单纯的列表升级为主控式的左侧结构树、右侧详情及员工的管理仪表盘。

### 3.5 国际化 (i18n) 与多语言支持 —— 缺乏动态适配与覆盖不全
**后端模块:** 动态 Schema / 字典 (Dictionary) / 全局多语言配置
**现状分析:** 尽管前端已有基础的 `i18n` 目录结构 (`zh-CN/index.ts` 等) 和静态映射，但在诸如表单设计器 (`ConditionBuilder`, `BaseEmptyState` 等) 及前端通用页面中仍存在不少硬编码的中文字符串。更深层次的缺口是，后端动态下发的视图模型 (Schema) 和业务字典等元数据缺乏与前端语言状态挂钩的联动机制，导致切换语言时这部分高频信息仍为单一语种。
**必须补齐功能:**
1. **硬编码文本清零与抽取**: 扫描全站视图与基础组件，将所有固定提示文字、表单标题、占位符沉淀至对应的 `json` 语言包中，在 Vue 组件中全面采用 `$t` 或 `useI18n` 进行替换。
2. **动态对象与 Schema 多语言解析对接**: 贯通对象注册表 (`ObjectRegistry`) 和元数据映射，使得基于 `fields API` 渲染出的 `label`, `placeholder`, `description` 能够触发动态翻译兜底逻辑。若后端下发多语言键值(如 `label_zh`, `label_en`)，则需在前端渲染前完成本地化清洗。
3. **用户偏好联动与请求改造**: 在全局顶部导航中提供优雅的语种切换入口；切换后无缝接驳到 `vue-i18n`，并利用拦截器使所有的 API 调用请求头带上合适的 `Accept-Language`，以让后端返回语言匹配的通知消息及预设系统数据。

## 4. 前端组件补齐的路径及方案

针对以上缺失与缺口，在 V4 UX 升级战略之上，不直接无脑硬编码这些页面，应结合现有“多态引擎”特性：

1. **全面实现双栏/多 Tab 详情架构 (详情页跃迁)**
   - 解决保险、租赁等需要组合信息展示的痛点。必须让 `DynamicDetailPage.vue` 升级支持包含 Tabs 解析的 schema。这能瞬间解锁生命周期明细、保险附属清单的视图能力。

2. **通用的“主子明细表 (SubTable)”组件化**
   - 采购申请单必然包含 N 条采购资产明细项。现有表单引擎主要支持平级 Record。需要补齐能在 `DynamicFormPage` 内部内嵌 `SubTable` 以批量新建/更新多行的组件。

3. **抽离通用的 Activity Timeline 业务轴**
   - 报修、采购到入库、工作流审批都需要这套 Timeline 去做流转可视化。

4. **建立前后端贯通的多语言管道 (i18n Pipeline)**
   - 打破纯前端静态翻译的局限。针对后端驱动的动态 Schema、列表等特征，核心要在 API 请求出口以及动态表单渲染引擎 (`LayoutAdapter` 等) 内置翻译适配钩子，保障在低代码页面构建时多语言无需冗余配置。

## 5. 建议执行顺序 (Sprint Rank)

- **Priority 1**: 通知铃铛及个人中心面板 (消除系统内的"聋哑"状态)。
- **Priority 1.5**: 全局多语言基建与静态文本抽离清洗 (在所有新增复杂页面铺开前规范好多语言机制，优先洗净已有组件内的硬编码字符串，避免返工带来技术债务)。
- **Priority 2**: 通用 SubTable 组件并对接资产生命周期(采购/维护/处置)。
- **Priority 3**: 深度重构 Dynamic Detail，落地复杂的租赁、保险组合 Tab 页面。
- **Priority 4**: 组织架构树等后台管理进阶视觉能力。
