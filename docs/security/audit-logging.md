# Audit Logging Documentation\n\n## Overview\n\nAudit logging is a critical component for security, compliance, and operational insights. It provides an immutable record of all significant actions within the GZEAMS system.\n\n## Purpose\n\n- **Security**: Detect and investigate unauthorized access or malicious activities.\n- **Compliance**: Meet regulatory requirements for data access and system changes.\n- **Accountability**: Track user and system actions to ensure proper governance.\n- **Troubleshooting**: Aid in diagnosing issues by providing a historical context of events.\n\n## Implementation Details\n\nThe audit logging service (`backend/apps/common/services/audit_service.py`) captures events and stores them for a configurable retention period.\n\n### Key Features\n\n- **Event Categorization**: Events are classified by type (e.g., `user_login`, `workflow_submitted`, `security_event`).\n- **Actor and Object Tracking**: Records who performed an action (`actor_id`, `actor_type`) and what object was affected (`object_type`, `object_id`).\n- **Contextual Details**: Additional `details` can be attached to each event for richer information.\n- **IP and User Agent**: Captures client IP address and user agent string for security analysis.\n- **Severity Levels**: Events are assigned a severity (low, medium, high) to prioritize review.\n- **Retention Policy**: Logs are retained for a configurable number of days (default: 90 days).\n- **Security Event Handling**: High-severity events trigger special logging and can integrate with external alerting systems.\n\n### Audit Event Types\n\n| Event Type | Description | Severity |\n|------------|-------------|----------|\n| `user_login` | User successfully logs in | Low |\n| `user_logout` | User logs out | Low |\
| `user_created` | New user account created | Medium |\
| `user_updated` | User account details updated | Medium |\
| `user_deleted` | User account deleted | High |\

| `workflow_updated` | Workflow definition updated | Low |\
| `workflow_deleted` | Workflow definition deleted | High |\
| `workflow_submitted` | Workflow instance submitted | Low |\
| `workflow_approved` | Workflow task approved | Low |\
| `workflow_rejected` | Workflow task rejected | Medium |\

\n---\n\n## API Usage\n\nThe `audit_logger` service can be imported and used directly within backend applications.\n\n### Log an Event\n\n```python\nfrom apps.common.services.audit_service import audit_logger\n\naudit_logger.log(\n    event_type=\'user_login\',\n    actor_id=request.user.id,\n    actor_type=\'user\',\n    action=\'User logged in successfully\',\n    ip_address=request.META.get(\'REMOTE_ADDR\'),\n    user_agent=request.META.get(\'HTTP_USER_AGENT\')\n)\n```\n\n### Log Workflow Actions\n\n```python\nfrom apps.common.services.audit_service import audit_logger\n\naudit_logger.log_workflow_action(\n    event_type=\'workflow_submitted\',\n    workflow_instance=instance, # WorkflowInstance object\n    actor_id=request.user.id,\n    action=\'Workflow submitted for approval\',\n    details={\'workflow_id\': str(instance.id), \'definition_code\': instance.definition.code}\n)\n```\n\n### Log Security Events\n\n```python\nfrom apps.common.services.audit_service import audit_logger\n\naudit_logger.log_security_event(\n    action=\'Suspicious login attempt\',\n    details={\'username\': \'bad_actor\', \'reason\': \'multiple failed attempts\'},\n    ip_address=\'192.168.1.100\'\n)\n```\n\n### Retrieve Audit Trail\n\n```python\nfrom apps.common.services.audit_service import audit_logger\n\n# Get audit trail for a specific workflow instance for the last 24 hours\naudit_events = audit_logger.get_audit_trail(\n    object_type=\'workflow_instance\',\n    object_id=workflow_instance_id,\n    hours=24\n)\n\nfor event in audit_events:\n    print(f\"[{event[\'timestamp\']}] {event[\'actor_type\']}:{event[\'actor_id\']} - {event[\'action\']}\")\n```\n\n### Generate Compliance Report\n\n```python\nfrom apps.common.services.audit_service import audit_logger\nfrom datetime import datetime, timedelta\n\nstart_date = datetime.now() - timedelta(days=30)\nend_date = datetime.now()\n\nreport = audit_logger.generate_compliance_report(start_date, end_date)\n\nprint(\"Compliance Report for the last 30 days:\")\nprint(f\"Total Events: {report[\'total_events\']}\")\nprint(f\"By Event Type: {report[\'by_event_type\']}\")\n```\n\n---\n\n## Configuration\n\nAudit logging can be configured via Django settings:\n\n- `AUDIT_LOGGING_ENABLED` (boolean): Enable/disable audit logging (default: `True`)\n- `AUDIT_LOG_RETENTION_DAYS` (integer): Number of days to retain logs (default: `90`)\n\n---\n\n## Best Practices\n\n- **Log granularly**: Capture sufficient detail for each event.\n- **Protect logs**: Ensure audit logs are secure and tamper-proof.\n- **Regular review**: Periodically review audit logs for anomalies.\n- **Integrate alerts**: Connect high-severity audit events to alerting systems.\n\n---\n\n## See Also\n\n- [Security Hardening Guide](/docs/security/hardening.md)\n- [Alert Rules Configuration](/docs/configuration/alert-rules.md)\n- [Monitoring API Documentation](/docs/api/monitoring-api.md)
